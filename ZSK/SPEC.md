## Behavior 规范（系统行为快照）

严格按照以下结构。

### 1. 系统行为摘要（结构化）

必须包含：
- LLM 使用分布
- skill 调用分布
- routing 分布
- cost / token 分布（如有）

### 2. 系统行为规律（核心输出）

3~10 条规则，格式：

```
**RULE-{ID}:**
- condition: {触发条件}
- behavior: {系统行为}
- observed effect: {观察到的效果}
```

### 3. 可执行系统规则（机器可用）

严格规则化，格式：

```
RULE-{DOMAIN}-{N}:
  IF {条件} THEN {行为}
  EFFECT: {影响}
```

### 4. 系统问题与偏差分析

必须列出：
- routing 偏差
- skill 失效或滥用
- token 浪费路径
- LLM 过度使用场景
- fallback 触发频率异常

### 5. ZSK 知识库写入块

```yaml
ZSK_VERSION: v{version}
TYPE: system_behavior_snapshot

INSIGHTS:
  - ...

RULES:
  - name: ...
    condition: ...
    action: ...

METRICS:
  ...

RISKS:
  - level: ...
    issue: ...
    impact: ...
```

### 6. 版本变化分析

- 变化点
- 系统稳定性评估
- 结构漂移检查

---

## 核心原则

不是"总结系统"，而是生成"可被系统执行的长期记忆结构"：

- 所有输出必须可以被 routing system、governance layer、future LLM manager 直接消费
- 规则必须机器可用，而非自然语言总结
- 禁止纯自然语言段落、禁止设计下个版本、禁止代码实现
